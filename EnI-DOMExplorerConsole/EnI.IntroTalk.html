DOM Explorer

What and Why

  • The goal

    To provide insight into the magically black box that is the browser so that developers can understand and diagnose issues in their app

  • Why is it needed?

    It’s needed because no sane person on this planet can rationalize how HTML & CSS will actually look once it’s rendered so naturally the only sane option is trial and error

What's left


Bucket #1: The little things that nag you and sometimes block you
  • Don’t support certain CSS constructs e.g. :before
  • Missing some simple feature copy image (etc.)
Bucket #2: Languages & Libraries
  • Developers don’t just write CSS they write LESS and SAS
  • Developers don’t just write vanilla they use bootstrap, jQuery and a truck load of stuff
Bucket #3: Innovation (especially in layout)
  • The DOM Explorer is stuck in the CSS2 box model world – the mobile web is not

JavaScript Console

What & Why

  • The goal:

    To provide the developer a playground to inspect and modify the state of their application.

    To provide the developer with actionable ‘to do list’ when things go wrong with their site.

    To enable the developer to instrument their applications so they can better understand where it all went wrong

  • Why is it needed:

    Cause ‘aint no one written perfect code

Never miss a console message

Turn on "Always record console messages"

Internet Options -> Advanced -> Browsing -> Always Record Console Messages (check)

Instrumenting Your Code

Use the console.* API to


  • console.time/timeEnd - to quickly time scenarios
  • performance.mark - to delineate scenarios in the profiler
  • console.takeHeapSnapshot - to take a snapshot
